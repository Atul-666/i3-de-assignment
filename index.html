Pipeline Architecture & Explanations

How Pagination Works

The OpenFDA API uses offset-based pagination via the skip and limit parameters.

  • Limit: The maximum number of records requested per API call (set to 100, which is the OpenFDA maximum without an authenticated API key).
  • Skip: The offset number of records to skip before returning data.
  • The Loop: The script uses a while loop. In each iteration, it fetches limit records, processes them, and then increments skip by limit (e.g., skip=0, then skip=100, then skip=200).
  • Termination: The loop stops cleanly if the target record count is reached, if an API call returns empty data, if a 404 (Not Found) error is raised (indicating no more data), or if the records returned are fewer than the limit.

Data Assumptions

Converting highly nested healthcare data to flat tabular data requires specific assumptions:

  • Multiple Drugs/Reactions: A single safety report often lists multiple drugs and multiple adverse reactions. To flatten this, we assume a semi-colon delimited concatenation is acceptable (e.g., Nausea; Headache).
  • Missing Identifiers: We assume records lacking demographic data (age, sex) should still be ingested, converting their missing values to "Unknown" rather than dropping the row entirely.
  • Sex Code Mapping: Assumed standard ICH mapping where 1 = Male, 2 = Female, and anything else (0 or missing) is Unknown.

Dataset Limitations (OpenFDA)

  • Voluntary Reporting Bias: The FAERS dataset relies largely on voluntary reporting by healthcare professionals and consumers. This means it cannot be used to calculate true incidence rates of adverse events (there is no denominator).
  • Causality: An adverse event report does not establish a causal relationship between the drug and the event; they are merely correlated in time.
  • Pagination Hard Limit: OpenFDA enforces a hard limit: the sum of skip + limit cannot exceed 25,000. For large-scale batch ingestions beyond 25K records, direct JSON bulk file downloads must be used instead of the API search endpoint.

Why Furosemide?

Furosemide (Lasix) was chosen deliberately — it is the primary diuretic prescribed to Bob Harmon, the post-discharge CHF patient analysed in Problem 1, on discharge from Riverside General Hospital.

In a real health system, this is exactly the kind of pharmacovigilance data a care team would reference when monitoring a post-discharge heart failure patient. Pulling its adverse event profile from OpenFDA creates a direct, meaningful connection between the two problems.

How AI Was Used

This pipeline was co-developed with Gemini. Here's what AI generated vs. what was verified independently:

  • AI generated: Initial pipeline structure, pagination loop, normalisation logic, retry handling, and the browser-side JavaScript reimplementation.
  • Independently verified: The 25,000 skip+limit hard cap was confirmed against the official OpenFDA documentation. The ICH sex code mapping (1=Male, 2=Female) was cross-checked against FAERS data standards.
  • Tested manually: Retry logic was verified by observing 429 responses in the browser console. Empty-data termination was confirmed by running beyond the available record count.
  • Judgment calls: Semi-colon concatenation for multi-drug reports, retaining unknown demographics rather than dropping — these were deliberate decisions, not AI defaults.
